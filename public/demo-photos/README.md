# Demo Photos — SiteCheck Block 4

These bundled placeholder images back the stub drone provider's `captureImage()`
call so the mission lifecycle works end-to-end without real drone hardware.

## How they're used

1. The stub `getDroneProvider().captureImage(missionId, waypointNumber)` in
   `src/lib/drone-provider.ts` returns one of these URLs deterministically
   based on the waypoint number.
2. `CaptureControls` POSTs that URL to the real photo upload endpoint
   (`/api/missions/[id]/waypoints/[number]/photos`), which fetches the bundle
   bytes and re-hosts them to the Supabase Storage `mission-photos` bucket.
3. `MissionCompleteRoute` then calls Claude vision (`src/lib/ai-vision.ts`) on
   each captured photo and persists the structured analysis into
   `mission_ai_analyses`.

This means the **same code path** is exercised in the demo as in production —
the only difference is the original image bytes.

## Replacing with real photos

To swap in real BMP photos for a customer demo:

1. Drop a JPEG at the same path/filename. Keep the names — `photo-1.jpg`,
   `photo-2.jpg`, etc. — so the deterministic stub picker still works.
2. Recommended size: 1024×768 or larger, JPEG quality ≥ 80.
3. Each photo should clearly depict the BMP category in its directory name so
   the Claude vision pass returns a believable analysis.
4. **Do not** commit photos containing identifying faces, license plates, or
   proprietary site details — anonymize first.

## Adding new categories

If you add a BMP category in `src/lib/cgp-bmp-categories.ts`, add a matching
directory here and update the `DEMO_PHOTOS` array in `src/lib/drone-provider.ts`
to point at the new files.
